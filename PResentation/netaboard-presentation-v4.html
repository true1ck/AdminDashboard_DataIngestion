STAGE 2 — LLM PILLAR SCORING
Per-Pillar Sentiment + Weighted NRI Aggregation

nriTotal = Σ(finalScore × pillarWeight)

03 — Message Bus

Kafka: The
Central Nervous System

Apache Kafka acts as the backbone between all services. Every microservice only talks to Kafka — never directly to another service. This means any service can be scaled, restarted, or upgraded independently without affecting others. Kafka also provides full replay capability — if the AI model is updated, all historical content can be reprocessed.

05 — Inter-Service Communication

How Services
Talk to Each Other

Different communication patterns are used depending on latency requirements, payload size, and whether the call is synchronous or event-driven. No service is exposed on a public port — all communication flows through the private VLAN or Kafka.

Kafka (Async Events)
Primary inter-service bus for all data flow
All content movement between pipeline stages. Decoupled — producer doesn't wait for consumer. Enables replay, backpressure handling, and horizontal scaling of any stage independently.
Connector → Pre-processor (raw.content topic)
Pre-processor → Enrichment (clean.content topic)
Enrichment → Embedding (enriched.content topic)
Alert Engine → Notification Service (alerts.triggered)
Dashboard corrections → Fine-tuning (model.feedback)
gRPC (Sync Calls)
Low-latency synchronous service calls
Used where a service needs an immediate response — e.g. the API server calling the LLM inference server for a real-time search query expansion, or the embedding server during live search. Protobuf serialization keeps payloads small and fast.
API Server → LLM Server A (query expansion, real-time summarize)
API Server → LLM Server B (on-demand embedding for search)
API Server → Qdrant (vector search queries)
API Server → Elasticsearch (full-text search queries)
REST / HTTP (Internal)
Admin, health checks, management APIs
Used for internal management endpoints, health checks, Kafka UI, MLflow, Kibana, and other monitoring tools. Also used by Celery workers to call the connector APIs to trigger new fetches. All over private VLAN only.
Celery Workers → Social API connectors (trigger fetch)
Monitoring → all services /health endpoints
MLflow → Training Server (metrics, model registry)
Kafka UI / Kibana → internal management
WebSocket (Push)
Real-time frontend dashboard updates
API Server maintains persistent WebSocket connections with each open dashboard session. When alerts.triggered fires on Kafka, the API server pushes the event to all relevant dashboard clients immediately — no polling required.
API Server → Frontend (alert fired events)
API Server → Frontend (live volume counters)
API Server → Frontend (new content notifications)
Kafka consumer → WebSocket relay → Browser

06 — What Was Missing & How We Fix It

Gaps, Risks &
Optimisations

A careful review of the initial architecture revealed 9 critical gaps that would cause failures in production. Every one is addressed below with a concrete fix and the tool/pattern used.

Critical
Single Point of Failure — API Server
SRV-01 as a single gateway means one crash takes the entire platform down. No failover, no health checks, no load distribution.
HAProxy in front of 2× FastAPI instances + Keepalived VIP for automatic failover. Nginx handles SSL termination.
Critical
No Secrets Management
WhatsApp API keys, Facebook tokens, JWT secrets, DB passwords — all would be in env files or config. One leaked file exposes everything.
HashiCorp Vault on SRV-01. All services fetch secrets at runtime via Vault agent sidecar. Secrets rotate automatically, never stored on disk.
Critical
No Dead Letter Queue on Kafka
If enrichment fails (malformed data, model timeout, OOM on GPU), messages are lost or cause infinite retry storms that block the pipeline.
DLQ topic per pipeline stage (e.g. enrichment.dlq). Failed messages route there after 3 retries. Separate DLQ consumer alerts ops and retries after cooldown.
High
No Circuit Breaker
If GPU server goes down, services continue hammering it with gRPC calls, causing cascading failures across the enrichment pipeline.
Tenacity (Python) circuit breaker on all gRPC calls to LLM servers. Falls back to a lightweight rule-based summariser when GPU is unavailable. Opens circuit after 5 consecutive failures.
High
No Service Discovery
With 8 servers and ~15 services, hardcoded IPs in configs will break every time a server restarts or IPs change. No health-aware routing.
Consul service mesh on private VLAN. Every service registers itself. gRPC clients resolve via Consul DNS (e.g. llm-server-a.service.consul:50051). Health checks built in.
High
No Backup / DR Strategy
PostgreSQL, Qdrant collections, Elasticsearch indices, MinIO objects — no backup plan means a disk failure = permanent data loss.
Daily pg_dump to MinIO (encrypted). Qdrant snapshot API → MinIO weekly. Elasticsearch ILM snapshots to MinIO. Off-site replication via rclone to a cold backup server nightly.
Optimise
Bloom Filter at Source
Currently deduplication runs inside the pre-processing service — meaning every item is ingested, parsed, and processed before duplicates are caught. Wastes GPU time.
Redis Bloom Filter at the connector level. Each connector checks a content hash before publishing to Kafka. 99.9% of duplicates filtered before they ever enter the pipeline.
Optimise
Data Validation Gate
Social media data is deeply inconsistent — null fields, encoding issues, truncated text. Without validation, corrupt records poison downstream models and indices.
Pydantic v2 schema validation on every Kafka message deserialization. Invalid records routed to validation.errors topic for manual review. Schema enforced via Kafka Schema Registry (Avro).
Optimise
Model Canary Deployment
When a new fine-tuned model version is pushed from SRV-08 to SRV-05/06, it replaces the old one instantly — no A/B testing, no safe rollout, no rollback mechanism.
MLflow Model Registry with Staging → Production promotion workflow. Canary: route 10% of traffic to new model version via feature flag in the LLM router. Rollback in <60s if quality drops.
